# Pre-Market Investor Pitch Decks

Use this skill when the user wants to create, improve, compare, or review an investor deck for a startup that still lacks mature traction, revenue history, or scaled distribution. This skill is for three raise types: `Equity`, `Equity + Token Warrant`, and `Token-only`.

The core idea is simple: at pre-market stage, the deck is not a report on performance. It is a structured argument for why the opportunity matters, why this team can execute, and how the raise reduces uncertainty.

1. Classify the raise type and note what is still unknown.
2. Read [references/shared-principles.md](references/shared-principles.md).

4. Build the output as a package, not just slides:
   - presentation deck

## Raise Selection

Choose `Equity` when:
- the token is not clearly required by the product
- jurisdiction, investor base, or issuance structure is still unclear
- the target audience is mainly traditional angels, seed funds, or VC
- the safest default is needed

Choose `Equity + Token Warrant` when:
- the product likely needs a token later, but token launch design is not final
- investors want equity protection now and conditional token exposure later
- the story must connect company value capture and network value capture without forcing a token sale too early

Choose `Token-only` when:
- the token is a real product primitive, not just a fundraising wrapper
- the team can defend token utility, distribution, vesting, compliance posture, and launch operations
- the audience is crypto-native and the project is ready for token-specific scrutiny

If the raise type is not locked, default to `Equity` and treat tokenization as a future design path rather than the starting fundraising assumption.

## Discovery Checklist

Capture these before drafting:
- product stage: idea, prototype, testnet, pilots, LOIs, waitlist
- investor type: angels, seed VC, crypto VC, strategics, community
- round shape: SAFE, note, priced round, side letter, warrant, SAFT, token purchase
- jurisdictions: company, token entity if any, investor geography
- what the capital buys: time, hires, compliance work, pilots, audits, testnet, GTM
- whether the deck is private, public, or likely to be forwarded widely

If the user does not provide these, make reasonable assumptions but label them as assumptions.

## Workflow

### 1. Define the job of the deck

Clarify whether the deck is meant to:
- win the next meeting
- support live pitching
- act as a read-alone send-ahead deck
- support DD or fundraising outreach

Do not force one document to do every job. A presentation deck and a send-ahead investor deck often need different density.

### 2. Build the proof ladder

For pre-market teams, replace missing traction with proof-of-demand and proof-of-execution:
- customer discovery interviews and pain evidence
- LOIs, pilots, waitlist quality, partner signals
- prototype, architecture, demo, security work, testnet
- relevant founder-market fit and hiring plan

Choose evidence that matches the raise type. For token scenarios, network, security, and incentive-readiness matter more than generic startup vanity metrics.

### 3. Build a risk-reduction narrative

Every strong pre-market deck should answer:
- what uncertainty exists now
- what this round funds
- what milestones remove the biggest risks
- what the company or network looks like at the next financing checkpoint

Use milestones, use of funds, and top risks as trust-building slides, not as legal boilerplate.

### 4. Keep structure tight

Prefer one idea per slide. Avoid decorative filler. A pre-market deck should feel crisp, causal, and disciplined.

When useful, show:
- why now
- beachhead market and ICP
- wedge into the market
- value capture logic
- proof ladder
- roadmap from current state to next financing state

### 5. Match language to legal sensitivity

This matters most for token-related raises.

Avoid language that sounds like:
- guaranteed returns
- passive income
- certain listing outcomes
- inevitable token price appreciation

Keep token claims tied to utility, mechanism design, user behavior, operating constraints, governance, distribution, and compliance posture. Treat the deck as an investor narrative, not as a substitute for legal documents or regulated disclosures.

### 6. Finish with the supporting package

For strong early-stage fundraising, prepare or recommend:
- terms summary
- cap table or pro forma table where relevant
- financial model or runway model
- DD checklist or data room outline
- token appendix, legal memo, or white paper pointers where relevant

## Review Mode

When reviewing an existing deck, focus on:
- wrong raise type for the actual business
- weak or missing proof ladder
- unclear ask, use of funds, or milestone logic
- token language that creates avoidable legal or credibility risk
- confusion between company value capture and token value capture
- missing appendix or DD readiness for sophisticated investors

Prioritize structural and risk issues over wording polish.

## Output Expectations

Good outputs from this skill usually include some or all of:
- recommended raise type
- target audience definition
- proposed slide order with 1-2 sentence purpose per slide
- what evidence belongs on each slide
- terms summary and appendix guidance
- top mistakes to avoid
- DD or supporting-document checklist
