# Snapshot And Controls Balance

## Use This Reference For

- deciding whether snapshot or controls should appear first
- separating "state" from "input"
- creating cleaner transitions between dashboard blocks

## Core Principle

Snapshot and controls may both belong near the top, but they should not feel like one undifferentiated slab.

They do different jobs:

- Snapshot tells the user what the dashboard currently says.
- Controls let the user challenge or change that state.

## Separation Techniques

When snapshot and controls sit close together, use at least two of these:

- larger vertical spacing
- different surface treatments
- a divider or rhythm break
- explicit headings
- a short context note that tells the user how the two zones relate

## When Snapshot Should Lead

Place snapshot above controls when:

- the page needs fast orientation
- the first screen should read like an executive summary
- the controls are secondary to the default story
- stakeholders want a "read first, tweak second" flow

## When Controls Should Lead

Place controls above snapshot when:

- every visible number is meaningless without a scenario choice
- the page acts more like a modeling tool than a report
- the user expects to set filters before reading anything

## Card Layout Guidance

- Use one row for short KPI cards on large screens.
- Use one row for related reading hints when the set is small and balanced.
- Collapse to fewer columns on narrower breakpoints without changing the conceptual order.
- Do not mix row and column patterns randomly within the same zone.

## What To Avoid

- snapshot cards directly touching control inputs with no visual pause
- headings that do not reveal the function of the block
- forcing every block into one column even when desktop comparison matters
- using space only after the user complains; build hierarchy proactively
